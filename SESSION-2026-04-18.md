## Known issues / what's still open

### High priority

- **No RTK Fixed ever achieved** in this garden with either the old or new
  GPS driver. Multi-constellation is live (12 sats), HDOP is good (0.5),
  NTRIP is delivering, but the receiver stays on Float. Likely environmental:
  base-station distance (crtk.net OUIL mountpoint), multipath from house,
  sky view. Not a driver bug.
- **Umeyama corrector needs motion**: needs ≥20 samples with ≥1.5 m of
  spread in both X and Y before it publishes `ALIGNED`. A pure-backward
  undock gives motion in one axis only, so the first autonomous transit
  after undock runs with identity `map→slam_map` and will target the wrong
  place. Workaround: drive a curve manually before hitting Start. Proper
  fix: option to run with yaw pinned to `dock_pose_yaw` and fit translation
  only (fast but less accurate).
- **Nav2 lifecycle race on startup** — if `mowgli-ros2` restarts after
  `mowgli-lidar`, `planner_server` times out its `global_costmap` activation
  at 60 s because `map→base_footprint` TF isn't yet complete. Two fixes
  applied so far (neither eliminates it completely): (a) rely on the second
  startup going through quickly since scans are already flowing, (b) manual
  `ros2 lifecycle set behavior_server configure/activate` after the
  `lifecycle_manager` bails. A proper fix is `wait_for_tf` gating the Nav2
  lifecycle_manager (existed on `feat/rtabmap`, not on `dev`).

### Medium

- **Old `garden_map.posegraph` confused slam_toolbox** — loaded a map from a
  prior session with a different dock pose, so the robot's initial pose in
  the loaded map was nonsensical. Wiped during this session; rebuild on next
  run.
- **`dock_pose_yaw`** in `install/config/mowgli/mowgli_robot.yaml` is
  currently `3.9794 rad` (228° compass). User measured **239° with phone

- **`imu_yaw`** same story — user measured **−2.781 rad**, was updated,
  stashed, not re-applied on this branch. Might cause residual heading error.
- **`install/docker-compose.yaml`** is checked into the branch now (new
  file); previously gitignored. The installer regenerates it per host — this
  will drift when a user reruns the installer. Need to either gitignore or
  move the lidar CMD change upstream.

### Low / future

## Key findings (verified this session)

1. **Wheels are accurately calibrated**, NOT 60 % over-counting as an
   earlier measurement suggested.
   - Test: manual-mow mode, backed up ~2 m, noted tick deltas ΔL=659 ΔR=628.
   - With firmware's `TICKS_PER_M=300`: wheel-computed distance = 2.145 m.
   - Tape-measured ground truth: **2.15 m** — 0.2 % error.
   - Implication: every "wheel slip" narrative from prior sessions was
     actually GPS noise (RTK Float jitter + bad heading validation).

2. **`heading_validated_` never demotes in FusionCore.** Once it's true
   (from a DUAL_ANTENNA dock heading publish or GPS_TRACK), it stays true
   for the run. Any bad early heading gets latched. Open: add a confidence
   decay so heading can be re-validated on reset.

3. **CIFASIS pattern is the right template** for the GPS-SLAM alignment.
   Their two-phase `NO_MEAS → ALIGNING → ALIGNED` state machine + weighted
   Umeyama on accumulated correspondences is strictly better than the
   running low-pass we had. Implemented today in `gps_slam_corrector`
   (commit `97cfe15`).

4. **Myzhar lidar driver has an upstream bug**: `count_subscribers(_scanTopic)`
   uses the unresolved relative topic name and always returns 0, so the
   driver reads the LiDAR at 10 Hz and never publishes. Patched via sed in
   the Dockerfile. Worth upstreaming.

5. **Docker Compose `command:` override** in `install/docker-compose.yaml`
   was silently invoking the OLD `ldrobotSensorTeam/ldlidar_stl_ros2` driver
   even after we built a new image. Always verify with
   `docker inspect ... Config.Cmd` after deploying a new image.

(abbreviated — commit history on the branch has the full story)

1. `feat/rtabmap` session: hit wall after wall — SLAM map "sunburst",

   wasn't converging without RTK Fixed.

   `feat/slam-toolbox-tuned`. Dropped the nav2-tuning cherry-pick (it had
   `use_rotate_to_heading: true` which caused pivot oscillation).
3. Dev's GPS container used the old `ros-kilted-ublox` serial driver, which
   can't configure an F9P on USB-CDC. Swapped in `sensors/gps/` from
   `feat/rtabmap` (aussierobots libusb driver).
4. Multi-constellation signals (`CFG_SIGNAL_*`) added to `ublox_dgnss.yaml`.
   NMEA went from GPS-only (`$GP*`) to mixed (`$GN*`), 12 sats, HDOP 0.5.
   Still no RTK Fixed — environmental, not driver.
5. `hardware_bridge` publishes `/imu/data` + `/wheel_odom` as `RELIABLE`
   (was `SensorDataQoS()` = BEST_EFFORT). FusionCore subscribes RELIABLE
   and was refusing every publisher → no sensor data → no odom.
6. Swapped LiDAR driver to **Myzhar `ldrobot-lidar-ros2`** for `bins: 455`
   (fixed scan length, slam_toolbox happy). Three build iterations:
   - Distro guard `FATAL_ERROR` → patched to `WARNING` via sed.
   - Entrypoint sourced wrong path → fixed to `/opt/ldlidar`.
   - Subscriber-count gating bug → patched via sed.
7. Rewrote `gps_slam_corrector` to the CIFASIS Umeyama pattern (commit
   `97cfe15`). Accumulates (slam, gps) correspondences, solves weighted
   SE(2) alignment once spread is sufficient, freezes the TF. Two build
   fixes (std_srvs linkage, param type) before it ran.
8. Bumped `local_costmap` update/publish frequency 5/2 → 10/10 Hz to kill
   sporadic `Costmap timed out waiting for update` errors.

## Useful debugging recipes

```bash
# Live state of Nav2 lifecycle
docker exec mowgli-ros2 bash -c "source /opt/ros/kilted/setup.bash && source /ros2_ws/install/setup.bash && \
  for n in behavior_server bt_navigator controller_server planner_server; do \
    echo -n \"$n \"; ros2 lifecycle get $n 2>&1 | tail -1; done"

# Corrector state (ACCUMULATING vs ALIGNED + residuals)
docker logs mowgli-ros2 --since 30s 2>&1 | grep -iE "ACCUMULATING|ALIGNED|gps_slam_corrector"

# Re-trigger alignment (force back to ACCUMULATING)
docker exec mowgli-ros2 bash -c "source /opt/ros/kilted/setup.bash && source /ros2_ws/install/setup.bash && \
  ros2 service call /gps_slam_corrector/reset_alignment std_srvs/srv/Trigger"

# TF chain
docker exec mowgli-ros2 bash -c "source /opt/ros/kilted/setup.bash && source /ros2_ws/install/setup.bash && \
  ros2 run tf2_ros tf2_echo map base_footprint"

  docker run --rm -v compose_mowgli_maps:/maps alpine sh -c "rm -f /maps/garden_map.*" && \
  docker start mowgli-ros2
```

## Suggested next steps (in priority order)

   `ALIGNED`. Watch logs for `ALIGNED: n=N yaw=X rms=Y m`. Once locked,
   `map→slam_map` is frozen and the SLAM pose on the map should be
   trustworthy.
2. If alignment takes too long operationally, add a **"fit translation only"**

   the user measured with phone compass — both got stashed during the
   branch switch.

   `feat/rtabmap`) to eliminate the startup race where `planner_server`
   times out waiting for `map→base_footprint`.
5. **Upstream the Myzhar driver subscriber-count patch** — it's a one-line
   fix (`count_subscribers(_scanTopic)` →

   base station too far? Pair with a closer CRTK mountpoint? Set up a local
